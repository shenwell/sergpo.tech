# Инструкция для агента (sergpo.tech)

Документ для ИИ-ассистента и людей: как правильно добавлять контент в Hugo и как убедиться, что деплой на GitHub Pages прошёл.

---

## 1. Заголовок Markdown-файла (YAML front matter)

### Правильный формат

Любая страница или пост в `content/` с метаданными **обязаны** начинаться так:

```yaml
---
title: "Заголовок страницы или поста"
date: 2026-05-08
draft: false
description: "Одно–два предложения для сниппета в выдаче и JSON-LD (рекомендуется для каждой страницы в content/)."
---
```

Дальше с новой строки идёт тело в Markdown/HTML.

Обязательно:

1. Первая строка файла — ровно `---` (открытие блока).
2. Поля — **YAML**, без префикса Markdown-заголовков: пишите `title:`, а не `## title:` и не `# title:`.
3. После полей — вторая строка `---` (закрытие блока). Без неё Hugo выдаст: `EOF looking for end YAML front matter delimiter`.
4. Тело документа — **после** второго `---`.

### Типичная ошибка (ломает CI)

Неправильно (так делать нельзя):

```yaml
---

## title: "я есть"
date: 2026-05-06
draft: false

Текст…
```

Здесь `## title:` воспринимается как Markdown, блок front matter не закрывается, сборка падает.

### Чеклист перед сохранением нового/изменённого `.md` в `content/`

1. Убедиться, что есть **два** разделителя `---` и между ними только YAML (ключ: значение), без `##` / `#` перед ключами.
2. Запустить проверку из корня репозитория:
  ```bash
   python3 scripts/verify-content-frontmatter.py
  ```
   Скрипт должен завершиться без вывода ошибок (exit code 0). Его же вызывает GitHub Actions перед `hugo build`.
3. Дополнительно (по желанию): `rg '^## title:' content/` — совпадений быть не должно.

### Где класть файлы

- **Статичные страницы** (im, memory, soul): `content/<язык>/pages/*.md`. В меню `hugo.toml` — `pageRef` вида `pages/im`, `pages/soul`, `pages/memory`.
- **Статьи блога:** `content/<язык>/write/<год>/имя-файла.md` (например `write/2026/…`). Раздел сайта остаётся `write`; год — вложенная папка.
- При переносе страницы старый адрес можно оставить через `aliases` в front matter (редирект на новый URL).

### Теги постов (`tags`) и вывод на странице

- Для **статей в секции `write`** список тегов задаётся в front matter массивом **`tags`**, например `tags: ["бег", "100км"]`. Дублировать те же слова хештегами в начале Markdown не нужно: при сборке перед `{{ .Content }}` подставляется блок со ссылками на страницы таксономии (шаблон **`layouts/write/single.html`**).
- Ссылки ведут на `/<язык>/tags/<urlize>/` (как принято в Hugo). Если тега нет в таксономии, страница терма всё равно может существовать при наличии постов с этим тегом.
- Для страниц **вне** `write` используется шаблон темы `_default/single.html` — там блок тегов не выводится.

### SEO: `description`, изображение, FAQ и HowTo (JSON-LD)

Тема **hugo-blog-awesome** подставляет `<meta name="description">` и соц-мета из поля **`description`** в front matter (если его нет — берётся общий текст из `hugo.toml` для языка). Для поиска и нейросетевых снимков страницы у каждой опубликованной страницы в `content/` должно быть **осмысленное уникальное** `description` (1–2 предложения), без пустой строки и без «заглушки».

Дополнительно в `<head>` подключается partial **`seo-jsonld.html`** (через `layouts/partials/custom-head.html`): для постов в секции **`write`** — `BlogPosting` и `BreadcrumbList`; при наличии данных в front matter — **`FAQPage`** и/или **`HowTo`**. Менять исходники темы не нужно.

**`image`** (необязательно) — путь к обложке для превью (Open Graph), например `/img/cover.png`. Не оставляйте ключ с пустым значением: либо заполните, либо удалите ключ.

**`faq`** (необязательно) — только если на странице **тем же текстом** даны вопросы и ответы. Формат:

```yaml
faq:
  - question: "Короткий вопрос?"
    answer: "Ответ, совпадающий с видимым на странице текстом."
```

Допустимы ключи `q` / `a` вместо `question` / `answer`.

**`howto`** (необязательно) — только если в теле статьи описаны те же шаги. Формат:

```yaml
howto:
  name: "Как сделать X"
  description: "Краткое вступление (по желанию)."
  steps:
    - name: "Шаг 1"
      text: "Текст шага, как на странице."
    - name: "Шаг 2"
      text: "…"
```

**Целостность:** не добавляйте `faq` / `howto` только ради разметки — иначе риск санкций в поиске и потери доверия в GEO. Опциональные поля с пустым значением не храните.

### Локальная сборка Hugo

Тема подключается как **Hugo Module** (`hugo.toml`). Команды `hugo mod get` и полная сборка сайта ожидают бинарий **Go** в `PATH`; иначе возможна ошибка `binary with name "go" not found`. В GitHub Actions Go ставится шагом **Setup Go** в `.github/workflows/hugo.yml`. На своей машине установите Go (или проверяйте результат только по успешному workflow после пуша в `main`).

## 2. Проверка деплоя после пуша в `main`

Пайплайн: **Deploy Hugo site to GitHub Pages** (файл `.github/workflows/hugo.yml`). После `git push origin main` не считать задачу выполненной, пока не подтверждён успешный прогон.

### Порядок действий для агента

1. Запомнить **полный SHA** последнего коммита после пуша, например `29e2b662883f27f9e96907185d654bd3fa23896e` (достаточно первых 7 символов для глаз, API отдаёт полный).
2. Подождать **30–90 секунд** (первый прогон обычно укладывается в эти рамки; при очереди раннера может быть дольше).
3. Запросить последние запуски workflow через GitHub API (публичный репозиторий, токен не обязателен, нужен заголовок `User-Agent`):
  ```text
   GET https://api.github.com/repos/shenwell/sergpo.tech/actions/runs?per_page=5
  ```
   Заголовок: `User-Agent: <любое осмысленное имя>` (без него API может отвечать 403).
4. В JSON взять первый элемент `workflow_runs[0]`, у которого:
  - `head_sha` совпадает с вашим коммитом (или с тем пушем, который проверяете);
  - `name` при желании сверить: `Deploy Hugo site to GitHub Pages`.
5. Интерпретация полей:
  - `status`: `queued` | `in_progress` | `completed`;
  - `conclusion`: пока `null`, пока не `completed`; при успехе — `success`, при ошибке — `failure` и т.д.
6. Если `status` ещё не `completed` — подождать **10–20 секунд** и повторить запрос того же `run` по URL из поля `workflow_runs[0].url` или снова списком `.../actions/runs?per_page=3`.
7. Сообщить пользователю итог: **успех** (`completed` + `conclusion: success`) или **провал** — дать ссылку `html_url` из объекта run на страницу логов в GitHub.

### Альтернатива: GitHub CLI

Если установлен `gh` и выполнен `gh auth login`:

```bash
gh run list --repo shenwell/sergpo.tech --workflow "Deploy Hugo site to GitHub Pages" --limit 3
gh run watch <RUN_ID> --repo shenwell/sergpo.tech --exit-status
```

### Ручная проверка

Открыть: [Actions репозитория](https://github.com/shenwell/sergpo.tech/actions).
